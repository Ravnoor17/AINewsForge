AINewsForge is an agentic LinkedIn content system built with LangGraph that autonomously fetches verified AI/ML news, generates technically sharp LinkedIn posts, fact-checks every claim against live sources, reviews post quality, and publishes to LinkedIn — all through a stateful multi-node graph pipeline.

The system is built around a directed graph where each node is a pure Python function operating on a shared AgentState. News is fetched from trusted AI sources via Tavily, a post is generated by Groq's llama-3.3-70b-versatile model following a strict structure, claims in the post are extracted and verified back against Tavily search results and source URLs, the post quality is scored by a second LLM reviewer pass, and finally the post is published to LinkedIn via the REST API after human confirmation.

The pipeline has two hard abort points. If fact verification scores below 70 out of 100, the pipeline stops and does not post. If post generation fails after three retries, the pipeline also stops. Quality review runs as a logging-only step and never blocks posting, giving visibility into post quality without gatekeeping.

The node architecture is fully modular. The news fetcher queries Tavily across three targeted queries covering model releases, open source frameworks, and research papers, filtering results to trusted domains including arxiv.org, huggingface.co, openai.com, anthropic.com, deepmind.google, github.com, mistral.ai, venturebeat.com, and techcrunch.com. The post generator enforces a five-part structure consisting of a sharp hook, technical context with benchmarks, engineering implications, an opinionated take, and relevant hashtags. The fact checker extracts three standalone factual claims from the post, searches each on Tavily independently, filters results by keyword relevance, runs LLM semantic verification for each claim returning SUPPORT, CONTRADICT, or UNCLEAR, rewrites contradicted or unclear claims using evidence before passing the post downstream, and checks source URL health as a non-blocking signal. The quality reviewer scores the post on hook sharpness, technical substance, practical engineering insight, clean structure, and hashtag relevance, logging issues and suggestions without blocking the pipeline.

The technology stack consists of LangGraph for agent orchestration, Groq llama-3.3-70b-versatile for post generation, fact claim extraction, semantic verification, and quality review, Tavily for news search and claim verification with a free tier of one thousand calls per month, the LinkedIn REST API version 202601 for publishing, and python-dotenv for configuration.

To set up the project, clone the repository and create a virtual environment. Install dependencies using pip install -r requirements.txt. Obtain a free Groq API key from console.groq.com and a free Tavily API key from tavily.com. For LinkedIn, create a developer app at developer.linkedin.com, add the Share on LinkedIn and Sign In with LinkedIn using OpenID Connect products, set the redirect URL to http://localhost:8000/callback, then run get_token.py to complete the OAuth flow and obtain your access token and person URN. The token expires every sixty days and get_token.py must be re-run to refresh it.

Configure your dot env file with client_id, client_secret, access_token, person_urn, groq_api_key, and TAVILY_API_KEY.

To run the agent in dry run mode without posting use python agent.py --dry-run. To run the full pipeline with LinkedIn posting use python agent.py.

The project structure has agent.py as the LangGraph graph and entry point, state.py as the shared AgentState TypedDict, and a nodes directory containing news_fetcher.py, post_generator.py, fact_checker.py, quality_reviewer.py, and linkedin_poster.py as individual pipeline nodes. get_token.py handles the one-time LinkedIn OAuth flow.

Every generated post follows a strict structure. The hook is one sharp specific sentence using a surprising fact, concrete number, or bold claim. The context section names the model, framework, paper, or company with benchmarks in four to five lines. The why it matters section explains the concrete engineering implication in four to five lines. The take section gives one opinionated engineering insight or prediction in two to three lines. Hashtags are three to five relevant non-generic tags at the end.

Planned improvements include token auto-refresh before the sixty day expiry, Langfuse tracing integration for full observability, a daily scheduler via APScheduler, Slack or email notification on post success or failure, and a post history log to avoid duplicate topics across runs.
